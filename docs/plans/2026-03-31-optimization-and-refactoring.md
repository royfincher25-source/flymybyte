# FlyMyByte — План оптимизации и рефакторинга

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Оптимизация, рефакторинг и расширение функционала FlyMyByte для Keenetic KN-1212 (128MB RAM, MIPS 580MHz)

**Architecture:** Модульная архитектура с разделением маршрутов по Blueprints, оптимизация производительности для embedded-устройств, улучшение надёжности и тестового покрытия.

## I. Критические проблемы

### 1. `routes.py` — 2178 строк, монолит
**Проблема:** Все маршруты в одном файле. Сложно тестировать, поддерживать, расширять.
**Решение:** Разбить на Blueprints:
- `routes_keys.py` — VLESS, SS, Trojan, Tor, Hysteria2
- `routes_bypass.py` — списки обхода, каталог, ipset
- `routes_service.py` — сервисные функции, бэкапы, обновления
- `routes_dns.py` — DNS monitor, DNS spoofing
- `routes_system.py` — логи, системные stats, auth

### 2. Избыточное логирование в `services.py`
**Проблема:** `logger.info()` на каждом шаге парсинга ключей (строки 44-68, 267-299). Создаёт огромный объём логов, забивает ротацию 100KB.
**Решение:** Заменить `logger.info` на `logger.debug` для отладочных сообщений. Оставить `info` только для ключевых событий.

### 3. Дублирование кода валидации CSRF
**Проблема:** `validate_csrf_token()` дублируется в `routes.py` и `app.py`.
**Решение:** Один источник истины в `core/utils.py`.

### 4. `dns_spoofing.py` — `_check_dnsmasq_status` использует `shell=True`
**Проблема:** `subprocess.run(['ps', '|', 'grep', 'dnsmasq'], shell=True)` —安全风险 + некорректный вызов.
**Решение:** Заменить на `subprocess.run('ps | grep dnsmasq', shell=True)` или лучше `subprocess.run(['pgrep', 'dnsmasq'])`.

### 5. `dns_spoofing.py` — `import threading` в конце файла (строка 553)
**Проблема:** Импорт после использования `threading.Lock()` в `__new__`. Может вызвать `NameError`.
**Решение:** Переместить `import threading` в начало файла.

## II. Оптимизации производительности

### 6. ThreadPoolExecutor `max_workers=2` — слишком мало
**Проблема:** В `routes.py:22` установлено 2 воркера. При одновременных запросах (обновление + проверка статуса) создаётся очередь.
**Решение:** Увеличить до 4 для KN-1212 (баланс между производительностью и памятью).

### 7. Обновление файлов — последовательная загрузка
**Проблема:** `service_updates_run()` загружает ~40 файлов последовательно. Каждый `requests.get()` блокирует поток.
**Решение:** Пакетная загрузка через `ThreadPoolExecutor` (3-4 параллельных запроса). Ускорит обновление на 60-70%.

### 8. `Cache.MAX_ENTRIES = 30` — можно оптимизировать
**Проблема:** LRU-кэш использует `list.remove()` — O(n) операция. При 30 записях это незаметно, но архитектура неэффективна.
**Решение:** Заменить `_access_order: List` на `collections.OrderedDict` — O(1) для LRU eviction.

### 9. `get_memory_stats()` читает `/proc/meminfo` при каждом вызове
**Проблема:** API endpoint `/api/system/stats` вызывает эту функцию каждый раз. При частых поллингах — лишние I/O.
**Решение:** Кэшировать результат на 5 секунд через `Cache`.

### 10. `check_service_status()` — subprocess на каждый сервис
**Проблема:** На странице `/keys` запускается `subprocess.run(['sh', init_script, 'status'])` для каждого из 5 сервисов. Даже с кэшем 30с — при загрузке страницы 5 subprocess'ов.
**Решение:** Заменить на `pgrep -f <process_name>` или проверку PID-файлов. Это быстрее и надёжнее.

## III. Рефакторинг архитектуры

### 11. Дублирование конфигурации: `app_config.py` vs `web_config.py`
**Проблема:** Два файла конфигурации с пересекающимся функционалом. `web_config.py` — shell-совместимый Python-файл, `app_config.py` — полноценный класс.
**Решение:** Объединить в один `core/config.py`. `web_config.py` оставить только как генерируемый shell-конфиг.

### 12. Singleton pattern без thread-safe `__new__` в `DNSSpoofing`
**Проблема:** В `dns_spoofing.py:68-75` используется `cls._lock if cls._lock else threading.Lock()` — создаётся новый Lock при каждом вызове до инициализации.
**Решение:** Инициализировать `_lock` на уровне класса: `_lock = threading.Lock()`.

### 13. `update_progress.py` — нет автосброса
**Проблема:** После завершения обновления статус остаётся `complete`. При следующем обновлении нужно вызывать `reset()`.
**Решение:** Автоматический сброс через TTL (например, 5 минут после завершения).

### 14. Hardcoded пути
**Проблема:** Пути `/opt/etc/...` захардкожены в десятках мест.
**Решение:** Вынести все пути в `core/constants.py` (файл уже существует, но пустой).

### 15. Шаблоны — дублирование навигации
**Проблема:** Каждый шаблон содержит собственные кнопки/формы без reusable компонентов.
**Решение:** Создать `templates/macros.html` с Jinja2 macros для повторяющихся элементов (карточки, кнопки, формы, алерты).

## IV. Улучшение надёжности

### 16. Отсутствие проверки свободного места перед обновлением
**Решение:** Добавить `os.statvfs('/opt')` перед загрузкой файлов.

### 17. `service_restart_router` — нет подтверждения
**Проблема:** Перезагрузка роутера одним POST-запросом без подтверждения.
**Решение:** Добавить модальное окно подтверждения на фронтенде + параметр `confirm=true` в POST.

### 18. Обработка `KeyboardInterrupt` при graceful shutdown
**Проблема:** DNS Monitor останавливается через `atexit`, но при `kill -9` поток остаётся.
**Решение:** Добавить signal handler для `SIGTERM`.

## VI. Идеи нового функционала (с учётом KN-1212)

### 1. Dashboard с виджетами реального времени
- Потребление RAM/CPU в реальном времени (обновление каждые 10с через SSE)
- Статус VPN-туннелей (ping до сервера)
- Количество доменов в ipset
- **Ограничение:** SSE легче WebSocket, не требует дополнительных зависимостей. Поллинг каждые 10с — минимальная нагрузка.

### 2. Планировщик задач (Cron UI)
- Визуальное управление crontab через веб-интерфейс
- Готовые шаблоны: обновление списков каждые 6ч, перезапуск dnsmasq ежедневно, бэкап еженедельно
- **Ограничение:** Использовать существующий `/opt/etc/crontab`, не запускать демонов.

### 3. Мониторинг VPN-соединений
- Ping до VPN-сервера с историей (last 24h, хранение в JSON-файле, не в БД)
- Автопереключение на резервный VPN при падении (failover)
- **Ограничение:** Хранить максимум 288 записей (каждые 5 мин × 24ч) — ~50KB.

### 4. Экспорт/импорт конфигурации
- Скачивание полного бэкапа одним архивом
- Восстановление из бэкапа через веб-интерфейс
- **Ограничение:** Использовать потоковую передачу, не загружать весь архив в RAM.

### 5. Тёмная/светлая тема с сохранением в cookie
- Переключатель темы в навбаре
- **Ограничение:** CSS-only переключение, без JS-фреймворков.

### 6. Быстрые действия на главной странице
- Кнопки «Перезапустить все сервисы», «Обновить списки», «Создать бэкап» прямо с dashboard
- **Ограничение:** AJAX-запросы, без перезагрузки страницы.

### 7. Лог-файл с фильтрацией и поиском
- Фильтр по уровню (ERROR, WARNING, INFO)
- Поиск по ключевым словам
- **Ограничение:** Показывать максимум 200 строк, серверная фильтрация.
