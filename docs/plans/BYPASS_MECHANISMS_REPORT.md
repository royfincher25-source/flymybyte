# Технический отчёт: Механизмы обхода региональных блокировок

**Дата:** 9 апреля 2026
**Версия:** 1.0

---

## 1. Типы блокировок и как они работают

### 🔴 Тип 1: DNS Pollution (DNS-подмена)

**Как работает:**
```
Клиент → DNS-запрос (chatgpt.com) → Провайдер DNS
                                          ↓
                                   Подменённый IP (ложный)
                                          ↓
                                   Клиент получает неправильный IP → сайт недоступен
```

**Решение:** Зашифрованный DNS через туннель
- **ss-tunnel** (Shadowsocks) — порт 5353, туннель к 8.8.8.8:53
- **stubby** (DNS-over-TLS) — порт 40500, зашифрованный DNS
- **dns2socks** — DNS через SOCKS5 прокси

---

### 🔴 Тип 2: IP-based блокировка (DPI)

**Как работает:**
```
Клиент → Трафик к chatgpt.com (реальный IP) → DPI система
                                                    ↓
                                          Определение заблокированного IP
                                                    ↓
                                          RST packet / DROP → соединение разорвано
```

**Решение:** Прозрачный прокси через iptables redirect
- Трафик перенаправляется на локальный порт прокси
- Прокси шифрует трафик → DPI не может определить целевой сайт

---

### 🔴 Тип 3: Geo-blocking (региональная блокировка)

**Как работает:**
```
Клиент → chatgpt.com → Сервер ChatGPT
                           ↓
                     Определение IP клиента → Россия/другая заблокированная страна
                           ↓
                     HTTP 403 / Redirect / "Not available in your region"
```

**Решение:** VPN прокси с IP другой страны
- Клиент → VLESS/SS/Trojan прокси → Целевой сайт
- Целевой сайт видит IP прокси-сервера (другая страна) → доступ разрешён

---

## 2. Технические механизмы обхода (сравнение)

### 🏆 Механизм 1: dnsmasq + ipset + iptables redirect (Классический)

**Архитектура:**
```
Клиент (192.168.1.100)
    ↓ DNS-запрос: chatgpt.com
dnsmasq (порт 53)
    ↓ правило: server=/chatgpt.com/127.0.0.1#5353
ss-tunnel (порт 5353)
    ↓ зашифрованный туннель
Внешний DNS (8.8.8.8:53)
    ↓ ответ: 204.155.148.100
dnsmasq добавляет IP в ipset: ipset=/chatgpt.com/gfwlist
    ↓
ipset gfwlist: 204.155.148.100
    ↓
iptables -t nat -A PREROUTING -m set --match-set gfwlist dst -j REDIRECT --to-port 1080
    ↓
ss-redir (порт 1080)
    ↓ зашифрованный трафик
Shadowsocks сервер (VPS)
    ↓
chatgpt.com (реальный доступ)
```

**Плюсы:**
- ✅ Автоматическое обновление IP в ipset при каждом DNS-запросе
- ✅ Защита от DNS pollution (зашифрованный DNS)
- ✅ Прозрачный для клиента (не нужно настраивать каждое устройство)
- ✅ Работает для всех устройств в сети

**Минусы:**
- ❌ Только TCP (UDP не работает через ss-redir)
- ❌ Зависит от доступности ss-tunnel
- ❌ Сложная настройка (3 компонента должны работать вместе)

**Где используется:**
- OpenWrt PassWall ✅
- OpenClash ✅
- FlyMyByte (частично — stubby вместо ss-tunnel)

---

### 🥈 Механизм 2: Xray routing (GeoIP/GeoSite)

**Архитектура:**
```
Клиент (192.168.1.100)
    ↓ любой трафик
Xray inbound (redirect/tproxy)
    ↓
Xray routing engine
    ↓ проверяет домен по geosite.dat (AntiZapret/Antifilter/v2fly)
    ↓ проверяет IP по geoip.dat
    ↓
┌─────────────────────────────────────────┐
│ domain:chatgpt.com → outbound:proxy    │
│ geoip:ru → outbound:direct             │
│ geosite:cn → outbound:direct           │
│ default → outbound:direct              │
└─────────────────────────────────────────┘
    ↓
VLESS/Trojan/SS сервер (VPS)
    ↓
Целевой сайт
```

**Плюсы:**
- ✅ Одно ядро — всё в одном (DNS + routing + proxy)
- ✅ Поддержка TCP + UDP
- ✅ GeoIP/GeoSite базы автоматически обновляются
- ✅ Reality/uTLS маскировка под обычный HTTPS

**Минусы:**
- ❌ Требует Xray ядро (~10-15MB RAM)
- ❌ Сложная конфигурация (JSON файлы)
- ❌ Не работает с dnsmasq/ipset (своя система маршрутизации)

**Где используется:**
- XKeen ✅
- PassWall ✅
- FlyMyByte (частично — только VLESS)

---

### 🥉 Механизм 3: Mihomo/Clash Rule-based routing

**Архитектура:**
```
Клиент (192.168.1.100)
    ↓ TUN mode (kmod-tun)
Mihomo (Clash Meta) ядро
    ↓ TUN interface забирает ВЕСЬ трафик
    ↓
Rule engine:
    - DOMAIN-SUFFIX,chatgpt.com → proxy
    - GEOIP,CN → direct
    - GEOSITE,gfwlist → proxy
    - MATCH → direct
    ↓
Proxy узел (VLESS/SS/Trojan/WireGuard)
    ↓
Целевой сайт
```

**Плюсы:**
- ✅ Полная прозрачность (TUN mode)
- ✅ Поддержка UDP + TCP
- ✅ Гибкие правила (domain, IP, GEO, process name)
- ✅ Встроенный DNS сервер с подменой ответов

**Минусы:**
- ❌ Требует ~50MB RAM
- ❌ Сложная настройка (YAML конфиги)
- ❌ TUN mode может конфликтовать с основным роутингом

**Где используется:**
- OpenClash ✅
- PassWall 2 ✅

## 3. Как FlyMyByte решает проблему региональных блокировок

### Текущая архитектура FlyMyByte:

```
Клиент (192.168.1.100)
    ↓ DNS-запрос: chatgpt.com
dnsmasq (порт 53)
    ↓
┌───────────────────────────────────────────┐
│ Для bypass доменов (vless.txt):           │
│   ipset=/chatgpt.com/unblockvless         │
│   DNS → обычный (8.8.8.8) ← МОЖЕТ БЫТЬ   │
│                        ПОДМЕНЁН! ❌       │
│                                           │
│ Для AI доменов (ai-domains.txt):          │
│   server=/chatgpt.com/127.0.0.1#40500    │
│   DNS → stubby (DoT) → ЗАЩИЩЁН ✅        │
│   НО: трафик НЕ через VPN ❌              │
└───────────────────────────────────────────┘
    ↓
ipset unblockvless: IP chatgpt.com
    ↓
iptables -t nat -A PREROUTING -m set --match-set unblockvless dst -j REDIRECT --to-port 10810
    ↓
VLESS proxy (порт 10810)
    ↓ зашифрованный трафик
VLESS сервер (VPS)
    ↓
chatgpt.com
```

### ⚠️ Проблема FlyMyByte:

**Два разных DNS-пути для одного домена:**

| Файл | DNS путь | Трафик | Результат |
|------|----------|--------|-----------|
| `ai-domains.txt` | stubby:40500 (зашифрованный) ✅ | Напрямую ❌ | DNS правильный, но трафик НЕ через VPN |
| `vless.txt` | обычный DNS ❌ | Через VLESS ✅ | DNS может быть подменён |

**Для полного обхода нужно:**
1. DNS через stubby (чтобы получить правильный IP)
2. Трафик через VLESS (чтобы IP был другой страны)

**Сейчас:** Нужно добавлять домен в ОБА списка — дублирование.

## 4. Как другие проекты решают эту проблему

### ✅ XKeen (лучшее решение):
**Один список → один механизм:**
- GeoSite базы (v2fly/AntiZapret/Antifilter) содержат домены
- Xray сам резолвит домены через свой DNS (зашифрованный)
- Xray сам маршрутизирует трафик (proxy или direct)
- **Нет дублирования** — домен только в одном месте

### ✅ PassWall / OpenClash:
**dnsmasq + chinadns-ng + ipset:**
- chinadns-ng резолвит через зашифрованный DNS (ss-tunnel)
- dnsmasq добавляет IP в ipset
- iptables redirect через прокси
- **Нет дублирования** — один список доменов

### ❌ FlyMyByte (текущее):
**Два списка → два механизма:**
- `ai-domains.txt` → stubby DNS (но нет VPN redirect)
- `vless.txt` → VPN redirect (но DNS может быть подменён)
- **Дублирование** — домен нужен в обоих списках

## 5. Как работает связка dnsmasq-full + ipset + ss-tunnel

### Разрешение доменов через защищённый DNS

В конфиге `dnsmasq-full` для доменов из списка указывается правило:
```
server=/домен/127.0.0.1#5353
```

При запросе такого домена `dnsmasq` не обращается к обычным DNS, а пересылает UDP-запрос на локальный порт `5353`. На этом порту слушает `ss-tunnel`, который инкапсулирует запрос в шифрованный туннель Shadowsocks и отправляет его на внешний DNS (например, `8.8.8.8:53`), получая корректный ответ.

### Автоматическое добавление IP в ipset

В том же конфигурационном файле используется директива:
```
ipset=/домен/gfwlist
```

После получения чистого IP-адреса от `ss-tunnel`, `dnsmasq-full` автоматически кэширует запись и добавляет разрешённый IP в ядерный набор `ipset` с именем `gfwlist` (тип `hash:ip`). Таким образом, статический список доменов динамически преобразуется в актуальный набор IP-адресов без участия пользователя.

### Перенаправление трафика через iptables

Ключевое правило NAT:
```bash
iptables -t nat -A PREROUTING -p tcp -m set --match-set gfwlist dst -j REDIRECT --to-port 1080
```

Если клиент пытается соединиться с IP, находящимся в `ipset`, `iptables` перехватывает пакет и перенаправляет его на порт `1080`, где работает `ss-redir`. `ss-redir` шифрует TCP-поток и отправляет его через удалённый Shadowsocks-сервер, обходя TCP RST-сбросы.

### Проблема DNS pollution и её решение

GFW/провайдеры систематически подменяют (отравляют) ответы DNS для заблокированных ресурсов, возвращая ложные или несуществующие IP. В данной схеме проблема решается принудительной маршрутизацией запросов к доменам из списка через `ss-tunnel`. Туннель защищает DNS-трафик от перехвата и подмены, гарантируя получение «чистых» IP от доверенных зарубежных серверов, которые затем автоматически попадают в `ipset` для корректного проксирования.

## 6. Как работает VLESS для обхода блокировок

### Технический механизм

VLESS использует радикально минимальный заголовок (25–50 байт) без магических чисел, специфичных опкодов или фиксированных структур. Весь трафик инкапсулируется в стандартный TLS 1.3.

**Рукопожатие полностью имитирует обычный HTTPS:**
- Стандартные cipher suites
- Валидные сертификаты (Let's Encrypt)
- SNI легитимных доменов (например, `cloudflare.com`)
- ALPN `h2`/`http/1.1`

После завершения TLS-рукопожатия данные VLESS передаются внутри зашифрованного поля `Application Data`. Для DPI-систем трафик неотличим от обычного веб-браузинга.

**Нейтрализация трёх методов детектирования:**
1. Fingerprinting протоколов — трафик выглядит как HTTPS
2. Статистический анализ — зашифрованный payload
3. Активное зондирование — fallback на обычный сайт

### Архитектура с CDN

```
Клиент → Cloudflare CDN → Origin-сервер (Nginx) → VLESS-сервер
```

- Клиент подключается к WebSocket-пути (например, `/api/v1/stream`) по порту 443
- Nginx на origin-сервере проксирует запросы с этого пути на локальный порт VLESS (`127.0.0.1:10000`)
- Запросы к корню сайта отдаёт как обычный веб-контент (fallback)
- VLESS-сервер парсит заголовок, извлекает целевой адрес/порт и транслирует данные между клиентом и целевым сайтом через outbound `freedom`

### Обход блокировки IP-адресов

Запуск VLESS на «голом» IP приводит к быстрому детектированию и внесению в чёрный список. Обход осуществляется за счёт интеграции с CDN (Cloudflare):
- Клиент подключается к общим IP-адресам инфраструктуры Cloudflare
- DPI-система видит только стандартный HTTPS-трафик к CDN
- Блокировка этих IP невозможна без масштабного нарушения работы интернета
- Реальный IP и географическое расположение origin-сервера полностью скрыты

## 10. Вывод

### Уникальные преимущества FlyMyByte:

1. **Python архитектура** — единственный проект на Python среди Bash/Lua/Go конкурентов
2. **Веб-интерфейс без LuCI** — работает на Keenetic где нет LuCI
3. **DNS-over-TLS (stubby)** — единственный с встроенным DoT
4. **OTA обновления** — единственный с встроенным механизмом обновления
5. **Оптимизация для 128MB RAM** — легче чем OpenClash/PassWall/v2rayA

### Главное преимущество перед конкурентами:

Простота использования через Web UI + Python архитектура для лёгкой поддержки и тестирования.

### Главный недостаток:

Меньше звёзд и сообщества, меньше протоколов, нет автообновления списков, дублирование доменов в двух списках (ai-domains + bypass).
