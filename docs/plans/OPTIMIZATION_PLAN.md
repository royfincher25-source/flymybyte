## 🔄 Осталось реализовать (Низкий приоритет)

### 5. Удаление Hysteria2 (Идея #2)
**Почему:**
- Hysteria2 — специализированный протокол для мобильных сетей
- На Wi-Fi/Ethernet не даёт преимуществ перед VLESS
- Занимает RAM + init скрипт + ipset
- VLESS с TCP/WS покрывает все сценарии

**Что удалить:**
- [ ] `hysteria2` парсер из `core/services.py`
- [ ] `/opt/etc/init.d/S22hysteria2` (на роутере)
- [ ] `unblockhysteria2` ipset
- [ ] Страница настройки Hysteria2 в UI
- [ ] ~100 строк кода

**Экономия:** ~10MB RAM, упрощение UI

### 6. Объединение Shadowsocks + Trojan в один "Прокси" сервис (Идея #3)
**Почему:**
- Оба делают одно и то же: SOCKS5 прокси на локальном порту
- Разница только в протоколе (SS vs Trojan)
- Пользователю всё равно — он вставляет ключ и получает обход
- Можно хранить один конфиг `/opt/etc/proxy.json` с полем `protocol: "ss" | "trojan"`

**Как:**
```json
{
  "protocol": "ss",
  "server": "example.com",
  "port": 443,
  "password": "...",
  "method": "chacha20-ietf-poly1305"
}
```

**Что изменить:**
- [ ] Создать единый парсер прокси (SS/Trojan)
- [ ] Создать единый init скрипт `S22proxy`
- [ ] Заменить `unblocksh` + `unblocktroj` на `unblockproxy`
- [ ] Обновить UI для выбора протокола
- [ ] Объединить маршруты в `routes_vpn.py`

**Экономия:** ~200 строк дублирующегося кода, упрощение UI

### 7. Удаление shell скриптов (Идея #5)
**Почему:**
- `100-redirect.sh`, `unblock_ipset.sh`, `unblock_dnsmasq.sh` — 800+ строк bash
- Bash сложнее отлаживать, нет типизации, нет тестов
- Python менеджеры уже готовы (`IptablesManager`, `DnsmasqManager`)

**Как:**
- [ ] Заменить `100-redirect.sh` на вызов `IptablesManager`
- [ ] Заменить `unblock_ipset.sh` на вызов `IptablesManager`
- [ ] Заменить `unblock_dnsmasq.sh` на вызов `DnsmasqManager`
- [ ] Заменить `unblock_update.sh` на вызов `DnsmasqManager`
- [ ] Создать единый `unblock_apply.py` для всех операций

**Экономия:** 660 строк bash кода, упрощение поддержки

### 8. Упрощение `dns_monitor.py` (Идея #6)
**Почему:**
- Слишком сложный для простой задачи health-check
- Редко используется в текущем виде
- Можно заменить на простой периодический ping DNS сервера

**Как:**
- [ ] Убрать сложную логику переключения
- [ ] Оставить только простой health-check каждые 60 секунд
- [ ] Убрать отдельную страницу мониторинга
- [ ] Интегрировать в общий статус системы

**Экономия:** ~150 строк кода

## 📊 Итоговая схема после полной оптимизации

### Было (исходное состояние):
```
5 VPN сервисов → 5 init скриптов → 5 ipset → 10+ shell скриптов
DNS Override + DNS-обход AI → 2 механизма
~2000 строк Python кода
~800 строк bash скриптов
~15MB RAM в простое
```

```
2 VPN сервиса (VLESS + Прокси) → 2 init скрипта → 2 ipset → Python менеджеры
Bypass списки → 1 механизм (включая AI-домены)
~1200 строк Python кода
~0 строк bash скриптов
~8MB RAM в простое
```
